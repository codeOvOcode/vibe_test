# 高校选课管理系统 - 编程实战（下）需求分析文档

## 一、项目背景

基于"学生选课基础处理工具"（第一题），使用 SpringBoot 3.x 框架进行功能升级，并设计简单前端页面用于 CSV 数据批量导入及数据展示。目标受众为 3 年以下经验的开发人员 / 应届生 / 实习生，技术栈要求简洁、分层清晰。

## 三、功能需求详细分析

### 3.1 后端功能需求

#### 3.1.1 实体类（Entity Layer）

| 实体 | 字段 | 说明 |
|------|------|------|
| `EnrollRecord` | `studentId` (String, S+6位数字) | 学生ID |
| | `courseId` (String, C+6位数字) | 课程ID |
| | `courseName` (String) | 课程名称 |
| | `courseType` (String) | 课程类型（公共课/专业课/选修课）— **新增字段** |

> **注意**：原 `EnrollRecord` 定义在任务描述中未包含 `courseType` 字段，但 CSV 导入数据格式（`S000001,C000001,Java程序设计,专业课`）包含课程类型，且在"编程实战"要求中明确需要分类功能。因此 `courseType` 必须作为实体字段。

#### 3.1.2 核心处理逻辑（Service Layer）

| 功能 | 输入 | 处理逻辑 | 输出 |
|------|------|----------|------|
| **去重** | `List<EnrollRecord>` | 按 `studentId + courseId` 组合去重（与 courseName 无关），保留首次出现记录 | 去重后的 `List<EnrollRecord>` |
| **排序** | `List<EnrollRecord>` | 先按 `studentId` 升序，再按 `courseId` 升序 | 排序后的 `List<EnrollRecord>` |
| **分类** | `List<EnrollRecord>` | 按 `courseType` 字段分类存储（公共课/专业课/选修课），支持：① CSV 导入时从数据中自动识别；② 手动标注修改 | `Map<String, List<EnrollRecord>>` 或分类后的列表 |
| **检索** | 关键词 + 类型 | 支持四种检索维度：`studentId`、`courseId`、`courseName`（模糊匹配）、`courseType`（精确匹配）。无结果时返回提示 "无匹配选课记录" | 匹配的 `List<EnrollRecord>` |
| **CSV解析** | CSV 文本字符串 | 解析格式：`studentId,courseId,courseName,courseType`，每行一条记录 | `List<EnrollRecord>` |
| **批量导入** | CSV文本（≥500条） | CSV解析 → 去重 → 排序 → 分类 → 存储/返回 | 处理后的完整列表 |

#### 3.1.3 性能要求

| 指标 | 要求 |
|------|------|
| 检索/排序响应时间 | ≤ 1 秒（1000 条以上记录） |
| 批量导入支持 | 单次 ≥ 500 条 |
| 实现策略 | 使用 HashMap 去重（O(n)），内存排序（O(n log n)），避免逐条数据库操作 |

#### 3.1.4 Controller 接口设计

| 接口 | 方法 | 路径 | 请求参数 | 返回 |
|------|------|------|----------|------|
| 批量导入 | POST | `/api/enrollment/import` | `csvData` (String, CSV文本) | 处理后的选课记录列表 (JSON) |
| 检索 | GET | `/api/enrollment/search` | `keyword` (String), `type` (String: studentId/courseId/courseName/courseType) | 匹配的选课记录列表 (JSON) |
| 获取样例数据 | GET | `/api/enrollment/sample` | 无 | 预置的样例选课记录列表 (JSON) |
| 按分类获取 | GET | `/api/enrollment/by-type` | `courseType` (String, 可选) | 按课程类型分类的记录 (JSON) |

### 4.2 并发风险分析

#### 核心并发问题：选课高峰期"超抢"（Overselling）

**场景描述**：热门课程容量有限（如 capacity=100），在选课开放瞬间数百名学生同时选课，可能出现：
- 事务 A 读取 capacity → 还有 1 个名额
- 事务 B 也读取 capacity → 还有 1 个名额（脏读）
- A 和 B 都认为有空位，都成功插入 enrollments
- 实际选了 101 人，超出容量 1 人

**根本原因**：`SELECT capacity → 判断 → INSERT enrollment` 三步操作不是原子的，存在 check-then-act 竞态条件。

#### 解决方案：数据库行级锁 + 乐观缓存

**推荐方案**（简单可行）：

```sql
-- 使用 SELECT ... FOR UPDATE 对课程行加排他锁
START TRANSACTION;
SELECT capacity FROM courses WHERE course_id = ? FOR UPDATE;
-- 检查剩余名额
SELECT COUNT(*) AS enrolled FROM enrollments WHERE course_id = ?;
-- 如果 enrolled < capacity，则插入
INSERT INTO enrollments (student_id, course_id, enroll_time) VALUES (?, ?, NOW());
COMMIT;
```

**原理**：
- `SELECT ... FOR UPDATE` 在事务内对目标行加排他锁（X锁）
- 其他并发事务的相同 SELECT 会被阻塞，直到当前事务提交
- 保证"查询容量 → 判断 → 插入"的原子性
- 实现简单，无需引入 Redis 等外部组件

**备选方案**（性能更优）：
```sql
-- 使用原子 UPDATE + 条件判断，一行 SQL 解决
UPDATE courses 
SET capacity = capacity - 1 
WHERE course_id = ? AND capacity > 0;
-- 检查 affected rows，若为 0 则选课失败
```

### 4.3 索引设计

#### enrollments 表索引

| 索引名 | 类型 | 字段 | 理由 |
|--------|------|------|------|
| PRIMARY KEY | 聚簇索引 | (student_id, course_id) | 联合主键，天然支持按学生查选课、按学生+课程去重 |
| idx_course_id | 普通索引（二级） | course_id | 高频查询：统计某课程选课人数、按课程检索 |
| idx_enroll_time | 普通索引 | enroll_time | 按选课时间范围查询（如统计某时段选课量） |
| idx_student_course | 联合索引 | (student_id, course_id) | 已由主键覆盖（MySQL InnoDB 中二级索引会存主键值） |

#### courses 表索引

| 索引名 | 类型 | 字段 | 理由 |
|--------|------|------|------|
| PRIMARY KEY | 聚簇索引 | course_id | 主键，课程唯一标识 |
| idx_course_type | 普通索引 | course_type | 按课程类型筛选（专业课/公共课/选修课）是高频查询 |
| idx_teacher_id | 普通索引 | teacher_id | 按授课教师查询其所有课程 |
| idx_type_capacity | 联合索引 | (course_type, capacity) | 优化"查询某类型课程中容量 > X"的场景 |

#### 设计理由总结

1. **主键即索引**：InnoDB 使用聚簇索引组织数据，主键选择直接影响查询性能。enrollments 的 (student_id, course_id) 联合主键天然覆盖按学生查询和去重场景。
2. **高频查询字段优先建索引**：course_id 在 JOIN 和 WHERE 中频繁出现，必须建索引。
3. **联合索引遵循最左前缀原则**：(course_type, capacity) 联合索引可同时服务 `WHERE course_type = ?` 和 `WHERE course_type = ? AND capacity > ?` 两种查询。
4. **避免过度索引**：enrollments 表中 student_id 单独建索引意义不大，因为联合主键的最左列就是 student_id，MySQL 可利用主键的最左前缀。
