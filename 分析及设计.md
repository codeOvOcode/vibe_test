# 高校选课管理系统 - 分析及设计文档

## 一、核心数据模型

### 1.1 完整 SQL 建表语句

```sql
-- ============================================================
-- 数据库创建
-- ============================================================
CREATE DATABASE IF NOT EXISTS enrollment_system
    DEFAULT CHARACTER SET utf8mb4
    DEFAULT COLLATE utf8mb4_unicode_ci;

USE enrollment_system;

-- ============================================================
-- 1. 学生表 (students)
-- ============================================================
CREATE TABLE students (
    student_id   VARCHAR(20)  NOT NULL COMMENT '学生ID，格式：S+6位数字',
    student_name VARCHAR(50)  NOT NULL COMMENT '学生姓名',
    department   VARCHAR(50)  DEFAULT NULL COMMENT '所属院系',
    grade        VARCHAR(10)  DEFAULT NULL COMMENT '年级',
    max_credits  INT          DEFAULT 30   COMMENT '最大可选学分',
    created_at   DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at   DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (student_id),
    INDEX idx_department (department),
    INDEX idx_grade (grade)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- ============================================================
-- 2. 教师表 (teachers)
-- ============================================================
CREATE TABLE teachers (
    teacher_id   VARCHAR(20)  NOT NULL COMMENT '教师ID，格式：T+6位数字',
    teacher_name VARCHAR(50)  NOT NULL COMMENT '教师姓名',
    department   VARCHAR(50)  DEFAULT NULL COMMENT '所属院系',
    title        VARCHAR(20)  DEFAULT NULL COMMENT '职称：教授/副教授/讲师',
    created_at   DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at   DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (teacher_id),
    INDEX idx_teacher_department (department)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- ============================================================
-- 3. 课程表 (courses)
-- ============================================================
CREATE TABLE courses (
    course_id    VARCHAR(20)  NOT NULL COMMENT '课程ID，格式：C+6位数字',
    course_name  VARCHAR(50)  NOT NULL COMMENT '课程名称',
    course_type  VARCHAR(20)  NOT NULL COMMENT '课程类型：公共课/专业课/选修课',
    capacity     INT          NOT NULL DEFAULT 30 COMMENT '课程容量',
    teacher_id   VARCHAR(20)  DEFAULT NULL COMMENT '授课教师ID',
    credits      INT          DEFAULT 2    COMMENT '学分',
    semester     VARCHAR(20)  DEFAULT NULL COMMENT '开课学期',
    created_at   DATETIME     DEFAULT CURRENT_TIMESTAMP,
    updated_at   DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (course_id),
    INDEX idx_course_type (course_type),
    INDEX idx_teacher_id (teacher_id),
    INDEX idx_course_type_capacity (course_type, capacity),
    CONSTRAINT fk_course_teacher FOREIGN KEY (teacher_id) REFERENCES teachers(teacher_id)
        ON DELETE SET NULL ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- ============================================================
-- 4. 选课记录表 (enrollments) — 学生与课程的多对多关联表
-- ============================================================
CREATE TABLE enrollments (
    student_id  VARCHAR(20)    NOT NULL COMMENT '学生ID',
    course_id   VARCHAR(20)    NOT NULL COMMENT '课程ID',
    enroll_time DATETIME       NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '选课时间',
    status      VARCHAR(10)    DEFAULT '正常' COMMENT '选课状态：正常/退选/待补选',
    score       DECIMAL(5,2)   DEFAULT NULL COMMENT '成绩',
    created_at  DATETIME       DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME       DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (student_id, course_id),
    INDEX idx_course_id (course_id),
    INDEX idx_enroll_time (enroll_time),
    INDEX idx_status (status),
    CONSTRAINT fk_enroll_student FOREIGN KEY (student_id) REFERENCES students(student_id)
        ON DELETE CASCADE ON UPDATE CASCADE,
    CONSTRAINT fk_enroll_course FOREIGN KEY (course_id) REFERENCES courses(course_id)
        ON DELETE CASCADE ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 1.2 表间关联关系

| 表A | 表B | 关系类型 | 关联字段 | 说明 |
|-----|-----|----------|----------|------|
| students | enrollments | **一对多** (1:N) | student_id | 一个学生可以有多条选课记录 |
| courses | enrollments | **一对多** (1:N) | course_id | 一门课程可以被多个学生选择 |
| students | courses | **多对多** (M:N) | 通过 enrollments 关联 | 学生选课：一个学生可选多门课，一门课可被多个学生选 |
| teachers | courses | **一对多** (1:N) | teacher_id | 一个教师可以教授多门课程 |
| courses | teachers | **多对一** (N:1) | teacher_id | 多门课程可以由同一位教师教授 |

### 1.3 实体关系图 (ER Diagram)

```
┌──────────────────┐          ┌──────────────────┐          ┌──────────────────┐
│    students      │          │    teachers      │          │     courses      │
├──────────────────┤          ├──────────────────┤          ├──────────────────┤
│ PK  student_id   │          │ PK  teacher_id   │          │ PK  course_id    │
│     student_name │          │     teacher_name │          │     course_name  │
│     department   │          │     department   │          │     course_type  │
│     grade        │          │     title        │          │     capacity     │
│     max_credits  │          │     created_at   │          │ FK  teacher_id   │
│     created_at   │          │     updated_at   │          │     credits      │
│     updated_at   │          └────────┬─────────┘          │     semester     │
└────────┬─────────┘                   │                    │     created_at   │
         │                             │ 1:N                │     updated_at   │
         │                             │                    └────────┬─────────┘
         │ 1:N                  ┌──────┴──────┐                     │
         │                      │   teaches   │                     │ 1:N
         │                      └──────┬──────┘                     │
         │                             │                            │
         ▼                             │                            ▼
┌──────────────────────────────────────────────────────────────────────────┐
│                              enrollments                                  │
│                          (学生-课程 多对多关联表)                            │
├──────────────────────────────────────────────────────────────────────────┤
│ PK,FK  student_id   ← 引用 students.student_id                           │
│ PK,FK  course_id    ← 引用 courses.course_id                             │
│        enroll_time                                                       │
│        status          (正常 / 退选 / 待补选)                              │
│        score                                                             │
│        created_at                                                        │
│        updated_at                                                        │
├──────────────────────────────────────────────────────────────────────────┤
│ PRIMARY KEY = (student_id, course_id)  — 联合主键确保学生不能重复选同一门课   │
└──────────────────────────────────────────────────────────────────────────┘

图例:
  ────▶  = 一对多关系 (1:N)
  PK     = 主键 (Primary Key)
  FK     = 外键 (Foreign Key)
  PK,FK  = 既是主键又是外键

关系说明:
  1. students ──(1:N)── enrollments ──(N:1)── courses
     → students 与 courses 之间是 多对多 关系，通过 enrollments 中间表解耦
  
  2. teachers ──(1:N)── courses
     → 一门课程只能由一位教师授课，但一位教师可以教多门课
```

### 1.4 关键设计决策

| 决策 | 说明 |
|------|------|
| **联合主键** `(student_id, course_id)` | 天然保证一个学生不能重复选同一门课，无需额外唯一约束 |
| **enrollments 作为中间表** | 将多对多拆解为两个一对多，符合数据库范式，查询灵活 |
| **teachers 与 students 无直接关联** | 学生通过 enrollments→courses→teachers 间接关联教师，符合现实语义 |
| **status 字段** | 支持选课生命周期管理（正常/退选），避免物理删除丢失历史数据 |
| **score 字段** | 成绩与选课记录绑定，一门课结课后录入，允许 NULL（未结课状态） |

## 二、并发风险分析

### 2.1 核心并发问题：选课高峰期"超抢"（Overselling）

#### 问题场景

热门课程容量有限（如 capacity=60），选课开放瞬间数百名学生同时选课：

```
时间线          事务A                       事务B
─────────────────────────────────────────────────────
T1          SELECT capacity=60      
T2                                     SELECT capacity=60
T3          判断: 60 > 已选59 ✓        
T4                                     判断: 60 > 已选59 ✓
T5          INSERT enrollment (第60人)  
T6                                     INSERT enrollment (第61人) ← 超抢！
```

**根本原因**：`SELECT → 判断 → INSERT` 三步操作不是原子的，在 check-then-act 窗口期内，其他事务可能也读到了旧的 capacity 值，导致超容量选课。

#### 其他并发风险

| 风险 | 场景 | 影响 |
|------|------|------|
| 重复选课 | 同一学生并发提交两次相同选课请求 | 联合主键冲突异常（可容忍，被数据库拒绝） |
| 脏读 | 事务A读到事务B未提交的选课记录 | 可能导致选课统计不准确 |
| 丢失更新 | 两个事务同时更新同一条选课记录状态 | 后提交的事务覆盖先提交的结果 |

### 2.2 解决方案：悲观锁（SELECT ... FOR UPDATE）

#### 方案描述

使用数据库行级锁，在事务内对课程行加排他锁，阻塞其他并发选课事务：

```sql
-- 选课事务伪代码
START TRANSACTION;

-- 对目标课程行加排他锁（X锁），阻塞其他并发事务的相同操作
SELECT capacity FROM courses WHERE course_id = ? FOR UPDATE;

-- 统计当前已选人数
SELECT COUNT(*) INTO @enrolled FROM enrollments 
WHERE course_id = ? AND status = '正常';

-- 检查名额
IF @enrolled < (SELECT capacity FROM courses WHERE course_id = ?) THEN
    INSERT INTO enrollments (student_id, course_id, enroll_time, status) 
    VALUES (?, ?, NOW(), '正常');
    COMMIT;
ELSE
    ROLLBACK;
END IF;
```

#### 方案原理

```
时间线          事务A                               事务B
────────────────────────────────────────────────────────────
T1          SELECT FOR UPDATE (获取X锁) ✓      
T2                                                SELECT FOR UPDATE (被阻塞...)
T3          判断名额 ✓                             (等待中...)
T4          INSERT ✓                              (等待中...)
T5          COMMIT (释放X锁)                      
T6                                                SELECT FOR UPDATE (获锁) → 读到新容量 → 判断失败
```

#### 方案优劣分析

| 维度 | 评价 |
|------|------|
| **正确性** | ✅ 严格保证不超抢，ACID 事务保证 |
| **实现难度** | ✅ 极低，一行 SQL 加锁即可 |
| **性能** | ⚠️ 单门课程的选课操作串行化，高并发下成为瓶颈 |
| **适用场景** | 适合选课系统——选课操作本身就需要串行判断容量，合理 |
| **死锁风险** | ⚠️ 如果按不同顺序锁定多门课程可能导致死锁，需统一加锁顺序（如按 course_id 排序） |

#### SpringBoot 实现示例

```java
@Transactional
public boolean enroll(String studentId, String courseId) {
    // JPA 的 @Lock(LockModeType.PESSIMISTIC_WRITE) 等价于 SELECT FOR UPDATE
    Course course = courseRepository.findById(courseId, LockModeType.PESSIMISTIC_WRITE)
            .orElseThrow(() -> new RuntimeException("课程不存在"));
    
    long enrolled = enrollmentRepository.countByCourseIdAndStatus(courseId, "正常");
    if (enrolled >= course.getCapacity()) {
        return false;
    }
    
    Enrollment enrollment = new Enrollment();
    enrollment.setStudentId(studentId);
    enrollment.setCourseId(courseId);
    enrollment.setEnrollTime(LocalDateTime.now());
    enrollment.setStatus("正常");
    enrollmentRepository.save(enrollment);
    return true;
}
```

### 2.3 备选方案：乐观锁（CAS 更新）

```sql
-- 使用原子 CAS 操作，一行 SQL 解决
UPDATE courses 
SET capacity = capacity - 1 
WHERE course_id = ? AND capacity > 0;
-- 检查 affected_rows: >0 成功, =0 失败（已满）
```

**对比**：乐观锁无阻塞，吞吐更高；但失败时需要客户端重试。选课高峰期重试会增加服务器压力，悲观锁更可靠。

## 三、索引设计

### 3.1 enrollments 表索引

| 索引名 | 类型 | 字段 | 设计理由 |
|--------|------|------|----------|
| `PRIMARY` | **聚簇索引** | `(student_id, course_id)` | 联合主键天然聚簇，按学生ID查询选课记录时减少回表；保证学生不能重复选同一门课 |
| `idx_course_id` | **二级索引** | `course_id` | 高频查询：统计某课程选课人数 `COUNT(*) WHERE course_id=?`，JOIN 查询依赖此索引 |
| `idx_enroll_time` | **二级索引** | `enroll_time` | 按时间范围查询选课记录（如统计某时段选课量、导出选课日志） |
| `idx_status` | **二级索引** | `status` | 筛选正常/退选状态的选课记录，配合 `course_id` 做复合条件查询 |

#### 索引使用场景

```sql
-- 使用 PRIMARY KEY (student_id 最左前缀)
SELECT * FROM enrollments WHERE student_id = 'S000001';

-- 使用 idx_course_id
SELECT COUNT(*) FROM enrollments WHERE course_id = 'C000001';

-- 使用 idx_status + idx_course_id (MySQL 会选择最优索引)
SELECT * FROM enrollments WHERE course_id = 'C000001' AND status = '正常';

-- 使用 idx_enroll_time (范围查询)
SELECT * FROM enrollments WHERE enroll_time >= '2025-02-20' AND enroll_time < '2025-02-21';
```

### 3.2 courses 表索引

| 索引名 | 类型 | 字段 | 设计理由 |
|--------|------|------|----------|
| `PRIMARY` | **聚簇索引** | `course_id` | 主键唯一标识，按课程ID精确查找时无需回表 |
| `idx_course_type` | **二级索引** | `course_type` | 高频查询：按课程类型筛选（专业课/公共课/选修课），如首页分类展示 |
| `idx_teacher_id` | **二级索引** | `teacher_id` | 按授课教师查询其所有课程（教师个人课表） |
| `idx_course_type_capacity` | **联合索引** | `(course_type, capacity)` | 优化复合查询：`WHERE course_type=? AND capacity > ?`，遵循最左前缀原则 |

#### 索引使用场景

```sql
-- 使用 PRIMARY KEY
SELECT * FROM courses WHERE course_id = 'C000001';

-- 使用 idx_course_type
SELECT * FROM courses WHERE course_type = '专业课';

-- 使用 idx_teacher_id
SELECT * FROM courses WHERE teacher_id = 'T000001';

-- 使用 idx_course_type_capacity（最左前缀）
SELECT * FROM courses WHERE course_type = '专业课';
-- 也使用 idx_course_type_capacity（完整匹配）
SELECT * FROM courses WHERE course_type = '专业课' AND capacity > 50;

-- JOIN 查询利用 PRIMARY KEY (courses) + idx_course_id (enrollments)
SELECT c.course_name, COUNT(e.student_id) AS enroll_count
FROM courses c
LEFT JOIN enrollments e ON c.course_id = e.course_id
GROUP BY c.course_id;
```

### 3.3 students 表索引

| 索引名 | 类型 | 字段 | 设计理由 |
|--------|------|------|----------|
| `PRIMARY` | **聚簇索引** | `student_id` | 主键，按学生ID精确查询 |
| `idx_department` | **二级索引** | `department` | 按院系统计学生、分学院选课分析 |
| `idx_grade` | **二级索引** | `grade` | 按年级筛选（如只允许大二学生选某类课） |

### 3.4 teachers 表索引

| 索引名 | 类型 | 字段 | 设计理由 |
|--------|------|------|----------|
| `PRIMARY` | **聚簇索引** | `teacher_id` | 主键 |
| `idx_teacher_department` | **二级索引** | `department` | 按院系查询教师列表 |

### 3.5 索引设计原则总结

1. **主键即索引**：InnoDB 使用聚簇索引组织数据，`enrollments` 的 `(student_id, course_id)` 联合主键天然覆盖按学生查询和去重场景，无需额外建 `student_id` 单列索引。

2. **高频查询字段优先**：`course_id` 在 JOIN、WHERE、GROUP BY 中频繁出现，必须建索引。联合主键已覆盖 `student_id`（最左前缀），无需重复建。

3. **联合索引遵循最左前缀**：`(course_type, capacity)` 可同时服务 `WHERE course_type=?` 和 `WHERE course_type=? AND capacity>?` 两种查询模式，减少索引数量。

4. **避免过度索引**：
   - 不在低基数字段建索引（如 `status` 只有 3 个值，但配合其他条件有用）
   - 不在频繁更新的字段建过多索引（影响写入性能）
   - `enrollments` 表插入频率最高（选课操作），索引数量控制在 5 个以内

5. **覆盖索引优化**：常用查询结果列若能包含在索引中，可避免回表。例如 `idx_course_type_capacity` 覆盖了课程类型筛选 + 容量检查的常见场景。
